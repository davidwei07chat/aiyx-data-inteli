## 1. 结论

AiYX DATA INTELI 的当前主线应从“通用舆情系统”收敛为：

> 面向品牌、公关、市场、运营和代理商的垂直行业品牌数据决策辅助平台。

第一阶段 MVP 不做所有行业，也不承诺强预测。先做三个示范行业：

| 行业 | MVP 示范对象 | 主要使用者 | 核心问题 |
| :--- | :--- | :--- | :--- |
| 美妆 | 珀莱雅 | 品牌、公关、市场、增长团队 | 新品反馈、成分口碑、竞品动态、营销机会 |
| 餐饮 | 海底捞 | 连锁运营、品牌、公关、区域管理团队 | 门店口碑、菜品反馈、服务痛点、区域风险 |
| 文旅 | 故宫博物院 | 景区、目的地营销、文旅运营团队 | 游客满意度、排队痛点、季节性热度、活动反馈 |

AI 的角色不是替客户做最终决策，而是：

- 汇总公开信息。
- 提取可验证信号。
- 给出多角度分析。
- 标注证据、置信度和局限。
- 形成辅助客户判断的建议。

最终决策仍由客户负责。

## 4. 产品定位

### 4.1 一句话定位

AiYX DATA INTELI 是一个基于公开信息采信、证据链整理和 AI 辅助分析的垂直行业品牌决策辅助平台。

### 4.2 不是什么

- 不是全网舆情大平台。
- 不是替客户做最终决策的自动决策系统。
- 不是承诺准确预测市场份额或竞品行为的预测产品。
- 不是靠堆 BI 图表卖复杂度的系统。
- 不是绕过平台限制的数据采集工具。

### 4.3 是什么

- 是品牌、公关、市场、运营团队每天查看的行业信号工作台。
- 是把公开信息整理成证据链、图表、洞察和建议的分析系统。
- 是能用示范品牌库快速展示价值的 Demo-Driven GTM 产品。
- 是可结合人工增强服务交付的商业化工具。

## 6. 数据源与合规边界

### 6.1 数据源原则

只使用以下来源：

- 公开网页和公开资讯。
- RSS 或公开订阅源。
- 客户授权或客户自行提供的数据。
- 标准 JSON、CSV、表格等手工导入数据。
- 合作方授权 API。

不提供：

- 绕过登录、验证码、风控或访问限制的能力。
- 未授权个人信息抓取。
- 对平台规则不可控数据的强依赖承诺。

### 6.2 公开信息采信流程

```text
公开信息发现
  ↓
来源记录与原始链接保存
  ↓
字段标准化
  ↓
去重与时间范围归一
  ↓
可信度评分
  ↓
AI 证据约束分析
  ↓
图表、报告、建议输出
```

### 6.3 可信度评分

每条信息和每个结论都应包含可信度字段：

| 维度 | 说明 |
| :--- | :--- |
| 来源类型 | 官方、媒体、平台用户、评论、转述、未知 |
| 原始链接 | 是否可追溯到公开链接或客户授权数据 |
| 时间新鲜度 | 是否处于当前分析窗口 |
| 样本量 | 是否足够支持趋势判断 |
| 多源一致性 | 是否被多个独立来源交叉验证 |
| AI 置信度 | 模型对结论稳定性的自评，仅作参考 |
| 人工校对状态 | 未校对、已抽检、已人工确认 |

展示方式：

- 高可信：可作为重点观察依据。
- 中可信：适合继续跟踪和补证据。
- 低可信：仅作参考，不建议直接行动。

## 7. AI 能力边界

### 7.1 AI 做什么

AI 负责：

- 信息摘要。
- 主题聚类。
- 情绪与风险倾向识别。
- 用户痛点归纳。
- 竞品动作整理。
- 多角度解释。
- 行动建议草案。
- 报告结构化生成。

### 7.2 AI 不承诺什么

AI 不承诺：

- 精准预测竞品何时行动。
- 精准预测用户是否流失。
- 精准预测市场份额变化。
- 自动给出唯一正确决策。
- 替客户承担业务判断责任。

### 7.3 决策辅助声明

所有 AI 建议应显示：

> 本建议基于当前公开信息和已采信证据生成，仅用于辅助判断。最终业务决策应由客户结合内部数据、预算、资源和风险承受能力自行决定。

## 8. 交互式动态数据图表

### 8.1 看板结构

每个示范品牌库包含：

- 总览指标区。
- 时间趋势区。
- 来源分布区。
- 主题/痛点聚类区。
- 竞品对标区。
- 风险信号区。
- 证据列表区。
- AI 建议区。

### 8.2 必备动态图表

| 图表 | 交互能力 | 决策价值 |
| :--- | :--- | :--- |
| 口碑/满意度趋势折线图 | 时间范围切换、来源筛选 | 观察口碑变化 |
| 声量柱状图 | 按日/周聚合、平台筛选 | 判断传播强度 |
| 情绪堆叠图 | 正/中/负切换 | 判断风险结构 |
| 痛点 Top N 条形图 | 点击查看证据样本 | 找改进优先级 |
| 主题热力图 | 行业关键词、时间窗口切换 | 找热点和机会 |
| 竞品雷达图 | 选择竞品维度 | 做对标沟通 |
| 来源可信度分布图 | 可信度筛选 | 避免被噪声误导 |
| 风险事件时间线 | 点击展开事件详情 | 复盘传播路径 |

### 8.3 图表设计原则

- 图表必须能回到证据列表。
- 图表必须支持时间范围筛选。
- 图表必须显示样本量。
- 图表必须显示更新时间。
- AI 建议必须能展开证据来源。
- 所有核心数据都必须提供动态、交互式图表展示，不允许只以静态文本或纯数据列表呈现。
- 图表交互优先使用轻量能力：筛选、悬浮提示、点击钻取、局部刷新、范围切换，避免为了炫技影响加载速度。
- 图表必须服务业务判断，避免纯装饰、纯渐变、纯“AI 感”的视觉堆叠。

### 8.4 数据列表与表格规范

所有数据列表必须按商业化数据产品标准设计：

| 要求 | 说明 |
| :--- | :--- |
| 行号 | 每行必须显示序号，便于会议沟通和报告引用 |
| 斑马纹 | 数据行颜色错开区分，提高长表扫描效率 |
| 冻结显示 | 序号列、第一列和表头必须冻结显示 |
| 排序 | 关键列支持升序/降序排序 |
| 筛选 | 必要字段支持下拉式数据选择，如行业、品牌、平台、情绪、风险等级、可信度 |
| 搜索 | 证据列表、评论列表和事件列表支持关键词搜索 |
| 列密度 | 支持紧凑/标准两种密度，默认紧凑但保证可读性 |
| 状态标签 | 风险、情绪、可信度、校对状态使用清晰标签，不用大面积纯色块 |
| 点击钻取 | 点击行可展开证据、原始链接、AI 解释和相关图表 |
| 空状态 | 无数据时说明原因，如数据源未更新、筛选过窄、样本不足 |

表格必须优先保证扫描、对比和定位，不做占空间的大卡片堆叠。

### 8.5 商业化视觉风格

UI 应呈现“专业、可视化、可信、轻快”的商业数据产品风格：

- 默认明亮主题。
- 使用多维配色表达数据类别，不使用单一纯色 AI 风格。
- 控制色彩面积，风险色只用于关键状态和提示。
- 图表、表格、筛选器和建议区要紧凑排布，减少无效留白。
- 视觉层级清晰：指标优先、趋势其次、证据可钻取。
- 避免大面积渐变、发光背景、装饰性图形和营销式 hero。
- 图标和按钮使用明确业务含义，如筛选、导出、刷新、钻取、对比。
- 首屏必须能看到关键指标、核心图表和最新风险/机会，不让用户先阅读说明文字。
- 页面交互要轻，加载要快；默认只加载当前视图必要数据，大表和证据列表分页或虚拟滚动。

### 8.6 性能与体验约束

- 首屏优先渲染关键 KPI、核心趋势和最新信号。
- 图表数据按时间范围和筛选条件增量加载。
- 大列表使用分页或虚拟滚动。
- 筛选变更只刷新受影响模块。
- 导出、AI 分析和报告生成可异步执行，并显示状态。
- 移动端以查看摘要和报告为主，复杂表格优先适配桌面端。

## 9. 竞品对标与反应信号模块

### 9.1 模块定位

竞品模块不做“竞品一定会怎么反应”的强预测，而是帮助客户回答：

- 竞品最近出现了哪些可验证动作？
- 这些动作可能说明什么？
- 我方与竞品的差距来自哪里？
- 哪些信号值得继续观察？
- 客户可以选择哪些应对动作？

输出必须保持证据约束和决策辅助边界：

> 系统提供竞品信号识别、对标分析和可选应对建议；是否采取行动、采取何种行动，由客户结合内部经营数据和资源自行决定。

### 9.2 竞品监测维度

| 维度 | 监测内容 | 示例 |
| :--- | :--- | :--- |
| 声量变化 | 提及量、互动量、增长速度 | 竞品近 7 天声量上升 42% |
| 内容动作 | 新品、活动、直播、联名、价格优惠 | 竞品集中发布新品种草内容 |
| 用户反馈 | 好评点、差评点、转粉原因 | 用户称“竞品更适合敏感肌” |
| 渠道动作 | 小红书、抖音、点评、微博、媒体报道 | 抖音直播频次明显增加 |
| 风险事件 | 负面扩散、投诉、服务问题 | 差评集中在配送和排队体验 |
| 营销节奏 | 节点、投放密度、KOL/KOC 动作 | 节日前 KOC 内容集中铺量 |

### 9.3 竞品反应信号分级

| 等级 | 信号 | 处理建议 |
| :--- | :--- | :--- |
| 观察信号 | 声量轻微上升、内容频率增加、互动小幅变化 | 继续观察，补充样本 |
| 明确信号 | 同类产品/活动上线、价格优惠、KOL 集中发布 | 进入竞品对标，评估是否响应 |
| 高风险信号 | 竞品借势我方负面、用户迁移讨论、舆论对比扩大 | 标记为重点事件，建议客户内部快速评估 |

### 9.4 对标维度

| 对标维度 | 问题 | 展示方式 |
| :--- | :--- | :--- |
| 口碑 | 谁的正负面反馈更集中？ | 情绪堆叠图、证据列表 |
| 声量 | 谁在当前周期更被讨论？ | 趋势折线、声量柱状图 |
| 产品/服务卖点 | 用户为什么选择对方？ | 主题聚类、痛点 Top N |
| 价格/活动 | 是否存在价格或促销压力？ | 活动时间线、对标表 |
| 渠道表现 | 哪个平台差距最大？ | 来源分布图 |
| 风险暴露 | 谁的问题更容易扩散？ | 风险事件时间线 |

对标结果不能只给“谁分数高”，必须说明差距来源，并允许用户点开证据样本。

### 9.5 反应建议输出格式

每条竞品反应建议使用固定结构：

```text
观察到的竞品动作：
可能含义：
证据来源：
可信度：
建议关注：
可选应对动作：
不建议动作：
需要客户内部确认：
```

示例：

```text
观察到的竞品动作：竞品 A 近 7 天在小红书集中发布“敏感肌修护”内容。
可能含义：竞品正在强化敏感肌用户心智，可能与近期换季护肤需求有关。
证据来源：公开小红书笔记、品牌官方账号内容、用户评论样本。
可信度：中高；多源一致，但样本仍集中在单一平台。
建议关注：未来 7 天互动增速、用户是否拿我方产品做对比。
可选应对动作：补充我方敏感肌证据内容，整理成分安全性 FAQ，观察是否需要 KOL 内容回应。
不建议动作：不建议直接攻击竞品或发布未经验证的对比结论。
需要客户内部确认：我方产品是否已有敏感肌测试、客服投诉数据和合规素材。
```

| :--- | :--- | :--- |

## 10. KPI 仪表盘

### 10.1 产品使用 KPI

| KPI | 定义 | MVP 目标 |
| :--- | :--- | :--- |
| Time to First Report | 从创建项目到生成首份报告的时间 | 小于 30 分钟 |
| Report Open Rate | 报告被打开的比例 | 试点客户大于 60% |
| Weekly Active Projects | 每周有数据更新或报告打开的项目数 | 稳定增长 |
| Insight Click Rate | 用户点击查看 AI 建议或证据的比例 | 大于 30% |
| Evidence Drill-down Rate | 用户从图表进入证据列表的比例 | 大于 20% |
| Feedback Rate | 用户对报告或建议反馈的比例 | 大于 20% |

### 10.2 数据质量 KPI

| KPI | 定义 | MVP 目标 |
| :--- | :--- | :--- |
| Source Health | 数据源成功更新比例 | 大于 90% |
| Evidence Coverage | 关键结论带证据链接的比例 | 100% |
| Sample Size Coverage | 图表显示样本量的比例 | 100% |
| High-confidence Insight Rate | 高可信结论占比 | 持续监控，不强求越高越好 |
| Low-confidence Warning Rate | 低可信结论被标注的比例 | 100% |

### 10.3 商业 KPI

| KPI | 定义 | MVP 目标 |
| :--- | :--- | :--- |
| Demo to Trial Rate | 看示范库后进入试用的比例 | 待验证 |
| Trial to Paid Rate | 试用转付费比例 | 试点期目标 20%-30% |
| Average Contract Value | 平均合同金额 | 4999-19999 元/月起步 |
| Paid Pilot Count | 付费试点客户数 | 首月 1-3 个 |
| Monthly Recurring Revenue | 月经常性收入 | 先验证，不做夸大预测 |
| Customer Expansion Rate | 单品牌扩展到多品牌或代理商项目比例 | 试点后观察 |

### 10.4 决策辅助 KPI

| KPI | 定义 | 说明 |
| :--- | :--- | :--- |
| Action Adoption Rate | 客户采纳报告建议的比例 | 由客户反馈或人工记录 |
| Risk Follow-up Rate | 风险信号被客户继续跟进的比例 | 衡量建议有效性 |
| Content Adoption Rate | 营销选题被采纳的比例 | 适合品牌和代理商 |
| Decision Confidence Score | 客户认为报告是否帮助判断 | 1-5 分反馈 |

### 10.5 竞品模块 KPI

| KPI | 定义 | 说明 |
| :--- | :--- | :--- |
| Competitor Signal Count | 当前周期识别到的有效竞品信号数 | 按观察、明确、高风险分级 |
| Competitor Signal Confidence | 竞品信号平均可信度 | 避免把噪声当趋势 |
| Competitive Gap Score | 我方与竞品在重点维度的差距评分 | 用于发现需要跟进的短板 |
| Evidence-backed Competitor Insight Rate | 带证据链接的竞品洞察比例 | MVP 目标 100% |
| Response Recommendation Adoption | 竞品应对建议被客户采纳的比例 | 衡量建议是否有行动价值 |

## 11. 商业模式

### 10.1 早期套餐

| 套餐 | 建议价格 | 目标客户 | 能力 |
| :--- | ---: | :--- | :--- |
| 试点版 | 999-1999 元/月，30 天 | 首批验证客户 | 单品牌日报/周报、基础看板、人工校对 |
| 品牌基础版 | 2999-4999 元/月 | 单品牌小团队 | 1 个品牌、3 个竞品、风险提醒、周报 |
| 品牌增长版 | 6999-19999 元/月 | 成长型消费品牌、连锁品牌 | 1 个品牌、5-10 个竞品、动态图表、AI 建议、月度复盘 |
| 代理商版 | 1.5-8 万元/月 | 公关公司、营销代理商 | 多品牌项目、白标报告、客户外发版、批量周报 |
| 私有部署版 | 5-50 万元/年 | 中大型客户 | 私有部署、权限、审计、备份、定制数据源 |

### 10.2 收入路径

第一阶段不靠大量低价订阅，靠以下组合验证商业价值：

- 示范品牌库获客。
- 低门槛付费试点。
- 人工增强周报。
- 代理商批量项目。
- 私有化部署。

### 10.3 为什么客户愿意付费

客户付费不是因为“有 AI”，而是因为：

- 少看大量公开信息。
- 更早发现风险。
- 更快整理竞品和用户反馈。
- 能拿报告向老板或客户汇报。
- 有证据链，能减少拍脑袋讨论。
- 能把日常舆情监控变成可复用资产。

## 12. GTM：Demo-Driven 品牌库

### 11.1 核心路径

```text
访客查看示范品牌库
  ↓
看到熟悉品牌的真实分析
  ↓
理解产品价值
  ↓
创建自己的品牌项目
  ↓
生成首份报告
  ↓
试点交付
  ↓
付费订阅或私有部署
```

### 11.2 销售话术

不要说：

> 我们是 AI 舆情监控系统。

应该说：

> 我们把公开信息整理成品牌每日决策看板：今天有什么风险、竞品有什么动作、用户在抱怨什么、市场团队可以关注哪些机会。AI 只做分析和建议，所有结论都有证据链接，最终决策由你们自己定。

### 11.3 首批获客方式

- 建立 30 个目标客户名单。
- 为 10 个目标品牌生成免费样例报告。
- 用示范品牌库约演示。
- 提供 30 天付费试点。
- 每周复盘客户是否真正使用报告。

## 14. 质量与可信度机制

### 13.1 三层质量控制

第一层：数据质量

- 来源链接必须保存。
- 时间范围必须明确。
- 样本量必须显示。
- 数据源失败必须标注。

第二层：AI 质量

- AI 输出必须引用证据。
- AI 输出必须结构化。
- AI 不得编造未采信事实。
- 低置信结论必须标注。

第三层：客户反馈

- 客户可以标记“有用/无用/需补证据”。
- 人工校对可以修正 AI 建议。
- 每周复盘哪些建议被采纳。

### 13.2 报告可信度呈现

每份报告必须包含：

- 数据时间范围。
- 数据来源列表。
- 样本量。
- 高可信结论。
- 中低可信观察。
- 不足与局限。
- 客户决策提示。

## 15. 阶段计划

### 14.1 第 1-2 周：可演示 MVP

- 固定三个示范项目：珀莱雅、海底捞、故宫。
- 准备公开样例数据或客户授权导入数据。
- 生成三个基础看板。
- 生成三份样例报告。
- 补证据链和可信度字段。

### 14.2 第 3-4 周：付费试点

- 找 10-15 个潜在客户演示。
- 争取 1-3 个付费试点。
- 记录客户是否愿意每天/每周看报告。
- 验证价格区间、报告结构和数据可信度要求。

### 14.3 第 2-3 个月：标准化交付

- 把人工校对沉淀为报告模板。
- 增加客户反馈闭环。
- 做代理商版雏形。
- 做私有部署基础包。

### 14.4 第 4-6 个月：扩展

- 每个行业从 1 个示范对象扩展到 3-5 个。
- 增加代理商白标报告。
- 增加私有部署客户。
- 评估是否恢复 AI 科技行业库。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
| :--- | :--- | :--- |
| 数据源不稳定 | 看板更新失败 | 数据源健康监控、手工导入兜底、失败标注 |
| AI 建议泛泛 | 客户不愿付费 | 行业模板、证据链、人工校对、反馈闭环 |
| 预测承诺过重 | 信任受损 | 只做分析建议，不做强预测承诺 |
| 范围扩张太快 | 交付失控 | MVP 三行业各一个示范库 |
| 客户认为只是报告 | 留存弱 | 增加动态图表、交互证据、每周复盘 |
| 合规风险 | 商业化受阻 | 只采信公开/授权数据，保留来源和限制说明 |

## 17. 最终建议

当前最优路径是：

1. 以本文档作为唯一主线文档。
2. MVP 做美妆、餐饮、文旅三个示范库，每个行业只做一个对象。
3. 数据源使用公开信息采信和客户授权导入。
4. AI 只做证据约束分析建议，不做最终决策和强预测。
5. 补齐 KPI 仪表盘、交互动态图表、可信度体系。
6. 用示范品牌库做 Demo-Driven GTM。
7. 用 30 天付费试点验证客户是否愿意付费。

一句话：

> 6 月方案给了商业化纪律，7 月方案给了品牌库销售力；v4 应把两者合并成一个更克制、更可演示、更容易成交的产品。
