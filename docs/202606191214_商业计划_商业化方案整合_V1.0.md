## 1. 执行摘要

TrendRadar 当前已经具备“热点抓取 + RSS 聚合 + 关键词过滤 + AI 分析 + 多渠道推送 + HTML 报告 + MCP 查询”的基础能力。它不应被包装成普通 RSS 阅读器，而应被定位为轻量级信息情报管线。

第一阶段商业化主线应聚焦：

> 品牌舆情与营销机会雷达。

核心价值不是“监控全网”，而是每天帮助品牌团队、公关团队、市场团队和营销代理商回答四个问题：

- 今天有没有品牌风险？
- 竞品在发生什么？
- 有没有可以借势的热点和内容选题？
- 市场团队今天应该做什么动作？

关键商业判断：

- 不强调“能抓很多信息”，强调“替客户少看信息、早做判断”。
- 不先做复杂 BI 大屏，先做每天能被打开的日报、晚报和异常提醒。
- 不让用户从零配置关键词，优先出售已经调好的品牌模板。
- 不优先打大型集团，先打中小品牌、公关公司、营销代理商和区域消费品牌。
- 不和 Meltwater、Brandwatch、Mention 正面对比覆盖范围，而对比部署成本、中文热榜、品牌场景模板、私有可控和 AI 助理化。

年收入 500 万人民币不应靠低价工具订阅实现，而应通过“软件 + 模板 + 人工增强 + 代理商批量交付 + 私有部署”的混合模式实现。

## 2. 收入目标与商业模式

### 2.1 500 万年收入目标拆解

| 收入来源 | 数量 | 单价 | 年收入 |
| --- | ---: | ---: | ---: |
| 品牌增长雷达版客户 | 20 个 | 4999 元/月 | 约 120 万 |
| 代理商版客户 | 5 个 | 3 万元/月 | 180 万 |
| 企业私有部署 | 10 个 | 15 万元/年 | 150 万 |
| 定制报告、品牌模板、人工增强服务 | 若干 | 项目制 | 50 万 |
| 合计 | - | - | 约 500 万 |

关键不是客户数量多，而是客单价足够高、交付足够标准化。第一年优先拿下 30-50 个高质量付费客户和 5 个代理商/服务商渠道。

### 2.2 产品套餐与定价

| 产品 | 建议价格 | 目标客户 | 说明 |
| --- | ---: | --- | --- |
| 开源版 | 免费 | 技术用户、潜在客户 | 自部署，提供基础采集、报告、推送、MCP，用于获客和建立信任 |
| 品牌基础监控版 | 1499-2999 元/月 | 单品牌小团队 | 1 个品牌、3 个竞品、每日舆情日报、负面提醒、周报 |
| 品牌增长雷达版 | 3999-6999 元/月 | 成长型消费品牌 | 1 个品牌、5-10 个竞品、营销机会、人工校对周报、月度复盘 |
| 代理商版 | 1.5-8 万元/月 | 公关公司、营销代理商 | 10-30 个品牌项目、多品牌管理、白标报告、客户外发版本 |
| 企业私有部署版 | 5-50 万元/年 | 中大型企业 | 私有服务器部署、专属数据源、权限、审计、备份和年度维护 |
| 人工增强服务 | 1-10 万元/月 | 高风险品牌、大型营销节点 | AI 自动监控 + 人工校对 + 策略建议 |

定价原则：

- 不过度竞争价格，强调价值而非便宜。
- 早期客户可提供 20-30% 折扣换案例。
- 年付可提供 10-15% 折扣。
- 代理商可提供 20-30% 渠道折扣。

### 2.3 可探索商业模式

除标准订阅、私有部署和人工增强外，可保留以下模式作为阶段性试验：

- **混合定价**: 基础月费 + 按热点数、AI 调用次数、推送次数计费。适合后期计量系统完善后引入。
- **咨询服务**: 报告 + 周度咨询 + 内容执行支持，价格 1-5 万元/月。适合高客单价试点。
- **联盟/分销**: 与营销服务商、公关公司、内容代运营合作，提供 API 和白标方案，分成 30-50%。
- **成果分成**: 与营销结果挂钩，理论客单价高，但效果追踪复杂，短期不作为主线。
- **免费增值**: 免费版获客，高级版付费；适合未来产品自助化成熟后扩展，不适合第一阶段承担收入目标。

## 3. 第一阶段产品边界

### 3.1 第一阶段只做什么

第一阶段只做一个主产品：

> 品牌每日舆情与营销机会助理。

标准交付物：

- 每日舆情日报。
- 重大负面风险提醒。
- 每周品牌/竞品/营销机会周报。
- 竞品动作追踪。
- 可借势热点和内容选题建议。
- 原始来源链接和证据链。
- 面向高客单价客户的人工校对版报告。

### 3.2 第一阶段不做什么

第一阶段暂不做：

- 全网舆情大平台。
- 复杂 BI 大屏。
- 面向所有行业的通用情报市场。
- App。
- 复杂支付系统。
- 重型多租户 SaaS。
- 与大型舆情平台拼数据覆盖。

这些能力可以后续补，但不应阻塞第一批收入。

## 4. 当前项目现状

### 4.1 当前定位

AIYXDATA-TRADAR 当前更准确的定位是：

> 面向个人和小团队的自动化热点情报系统。它从热榜平台和 RSS 源采集信息，通过关键词与标签规则筛选，再由 AI 生成研判，最后以 HTML 报告和协作工具消息的形式分发。

商业化后应升级为：

> 面向品牌、公关、市场和代理商的每日舆情与营销机会助理。

### 4.2 已具备能力

- **热榜数据抓取**: 支持今日头条、百度热搜、华尔街见闻、澎湃新闻、Bilibili、财联社、凤凰网、贴吧、微博、抖音、知乎等来源。
- **RSS/Atom 聚合**: 支持自定义 RSS/Atom 源，支持新鲜度过滤。
- **关键词与标签系统**: 支持普通关键词、正则、别名、组别名、必须词、排除词、显示条数限制和 `@tags` 分类标签。
- **报告模式**: 支持 `incremental`、`current`、`daily` 三种模式。
- **AI 分析**: 通过 LiteLLM 接入模型服务，支持 OpenAI 兼容接口和多模型提供商。
- **多渠道通知**: 支持飞书、钉钉、企业微信、Telegram、邮件、ntfy、Bark、Slack、Generic Webhook。
- **HTML 报告与配置中心**: 已有可视化报告、配置编辑和 API 读写能力。
- **MCP 查询方向**: 具备向 AI Agent 暴露新闻和趋势数据能力的基础。

### 4.3 当前商业化缺口

- **产品定位缺口**: 仍偏技术工具，需要转成品牌/公关/市场语言。
- **安全缺口**: 商业化前必须补齐认证、权限、密钥保护、审计和私有部署安全基线。
- **数据源稳定性缺口**: 第三方热榜和 RSS 存在不可控风险，需要健康监控、失败降级和多源冗余。
- **多租户与账号缺口**: 当前更适合单实例，需要项目、客户、品牌、团队权限模型。
- **商业报告缺口**: 报告应从信息摘要升级为决策建议、风险评分、机会评分、证据链和行动清单。
- **运营与交付缺口**: 需要模板库、交付 SOP、试点流程、客户成功和反馈闭环。

## 5. 商业化方向选择

### 5.1 推荐主方向

第一阶段主战场：

1. 中小消费品牌舆情雷达。
2. 公关公司与营销代理商。
3. 区域连锁与本地生活品牌。
4. 创始人 IP 与个人品牌。
5. 品牌营销机会雷达。

这些客户共同特点是：每天需要看信息、有明确营销预算、需要对老板或客户交报告、愿意为减少人工和降低风险付费。

### 5.2 暂缓方向

暂缓方向包括：

- 投研雷达。
- 政策监管雷达。
- 电商选品雷达。
- HR/招聘雷达。
- 医疗/健康雷达。
- 城市运营雷达。
- 创作者选题雷达。
- MCP 数据服务商业化。
- 企业知识库和行业数据集。

这些方向都有潜力，但需要垂直专业知识、额外数据源、不同模板和更复杂交付。建议先验证品牌主线，再逐步扩展。

### 5.3 阶段性扩展路径

- **0-6 个月**: 聚焦品牌舆情与营销机会雷达，验证商业模式，积累客户案例。
- **6-12 个月**: 扩展代理商版和私有部署版，提高客单价，建立渠道。
- **12-24 个月**: 探索投研雷达、城市运营、跨境电商等垂直扩展。

## 6. 客户群体与细分策略

### 6.1 优先客户

| 客户类型 | 价值 | 成交理由 |
| --- | --- | --- |
| 公关公司和营销代理商 | 高客单价、可批量复制 | 一次代理多个品牌，需要日报、周报、客户外发报告 |
| 成长型消费品牌 | 高需求、中等周期 | 有投放预算，需要看竞品、风险和热点 |
| 餐饮和本地生活连锁 | 本地热点敏感 | 需要区域舆情、门店事件和营销选题 |
| 文旅、活动、商业地产 | 营销节点明确 | 需要活动传播和热点借势 |
| 创始人 IP 和知识付费品牌 | 声誉敏感 | 需要个人品牌风险和选题建议 |

### 6.2 后续客户

- **企业级客户**: 大型集团、上市公司、国企/央企，客单价高但采购周期长，需要合规和私有部署。
- **中小企业客户**: 电商卖家、SaaS 创业公司、本地服务商，采购周期短但价格敏感。
- **个人用户**: 自媒体创作者、投资者、学生研究者，基数大但客单价低，不承担第一阶段收入目标。
- **行业协会/媒体**: 可作为数据合作和影响力渠道。

### 6.3 不适合第一阶段销售的客户

- 只是想看新闻的个人用户。
- 没有品牌投放预算的小商家。
- 要求全网覆盖的大型集团。
- 合规审批很慢的国企/央企。
- 只愿意买低价工具、不需要报告服务的用户。

## 7. 报告内容与交付形式

### 7.1 标准报告结构

推荐采用决策导向结构：

```text
今日结论（一句话）
├─ 风险评分（1-10）
├─ 机会评分（1-10）
├─ 建议动作（3-5 条）
├─ 证据链（原始链接）
└─ 下一步（明天关注什么）
```

同时支持品牌对标结构：

```text
我们 vs 竞品 A vs 竞品 B
├─ 声量对比
├─ 情绪对比
├─ 内容策略对比
├─ 用户反馈对比
└─ 可学习点
```

以及行动清单结构：

```text
市场团队今天应该做什么
├─ 紧急动作（今天）
├─ 重要动作（本周）
├─ 观察动作（持续关注）
└─ 不需要做的（避免踩坑）
```

### 7.2 内容模块

- 风险等级。
- 主要负面信号。
- 竞品动态。
- 营销机会。
- 热点借势建议。
- 内容选题建议。
- 原始证据链。
- 下一个观察窗口。

### 7.3 交付形式

第一阶段优先：

- HTML 报告。
- 飞书、企微、邮件推送。
- 人工校对 PDF 或可转发版周报。

后续可探索：

- PDF 正式报告。
- 实时数据 Dashboard。
- Slack/Teams 集成。
- 微信小程序。
- 语音播报。
- 1-3 分钟短视频总结。

## 8. 目标系统架构与产品化改造

### 8.1 商业化目标架构

商业化版本应从“单实例配置工具”演进为“项目化、多客户、模板驱动”的系统：

```text
客户/项目层
├─ 品牌项目
├─ 竞品配置
├─ 关键词模板
├─ 数据源模板
└─ 报告模板

采集与分析层
├─ 热榜抓取
├─ RSS 聚合
├─ 关键词匹配
├─ 风险评分
├─ 机会评分
└─ AI 策略建议

交付层
├─ 日报
├─ 周报
├─ 异常提醒
├─ 客户外发报告
└─ API/MCP

运营层
├─ 客户管理
├─ 模板管理
├─ 数据源健康
├─ 成本统计
└─ 反馈闭环
```

### 8.2 P0 安全与可商用底座

商业化前必须优先完成：

- 登录认证。
- 管理员账号。
- API 密钥不暴露到前端。
- 基础权限控制。
- 服务器路径和配置文件访问保护。
- 操作审计日志。
- 数据备份和恢复。
- 部署文档和环境变量模板。

### 8.3 P1 品牌模板系统

需要支持：

- 品牌名称。
- 竞品列表。
- 品牌别名。
- 风险关键词。
- 机会关键词。
- 行业热点词。
- 排除词。
- 报告模板。
- 推送渠道。

这部分是商业化交付效率的核心。客户不应从空白配置开始，而应从行业模板开始。

### 8.4 P1 项目化管理

需要新增：

- 客户项目。
- 品牌项目。
- 多品牌配置。
- 项目级报告。
- 项目级推送。
- 项目级数据源健康。

### 8.5 P1 信号评分

建议引入：

- 风险评分。
- 机会评分。
- 竞品活跃度。
- 声量变化。
- 异常弱信号。
- 证据可信度。

### 8.6 P1 报告模板升级

报告从“信息列表”升级为“决策报告”：

- 今日摘要。
- 风险红黄绿灯。
- 竞品动态。
- 营销机会。
- 内容建议。
- 证据链。
- 推荐动作。
- 人工点评区。

### 8.7 P2 多租户与商业后台

第二阶段再做：

- 多租户账号。
- 团队协作。
- 套餐与订阅。
- 用量统计。
- 成本核算。
- 客户成功看板。
- 代理商白标。

## 9. 部署方案

### 9.1 当前 Docker 部署

当前项目适合用 Docker Compose 快速部署演示版和私有化试点版。需要稳定以下内容：

- 应用容器。
- 配置目录挂载。
- 输出目录挂载。
- 日志目录挂载。
- 数据库持久化。
- Nginx 反向代理。
- TLS 证书。

### 9.2 私有部署推荐架构

```text
客户服务器
├─ Nginx / HTTPS
├─ TrendRadar Web/API
├─ Scheduler / Worker
├─ SQLite 或 PostgreSQL
├─ config/
├─ output/
├─ logs/
└─ backup/
```

私有部署必须提供：

- 安装脚本。
- 环境变量模板。
- 默认管理员初始化。
- 健康检查接口。
- 日志路径说明。
- 备份恢复方案。
- 升级说明。

### 9.3 环境变量建议

应补齐：

- `APP_ENV`
- `APP_SECRET_KEY`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `BASE_URL`
- `DATABASE_URL`
- `CONFIG_DIR`
- `OUTPUT_DIR`
- `LOG_DIR`
- `AI_PROVIDER`
- `AI_API_KEY`
- `AI_MODEL`
- `ENABLE_MCP`
- `ENABLE_AUTH`

### 9.4 GitHub Actions 部署

后续可加入：

- Lint。
- 测试。
- Docker build。
- 镜像推送。
- 远程部署。
- 回滚。
- 安全扫描。

## 10. 开发与销售路线图

### 阶段 0: 安全整改与可演示版本

目标：从“能跑”变成“敢演示”。

- 补登录认证。
- 保护配置和密钥。
- 固化 Docker 部署。
- 准备 3 个品牌样例。
- 生成可外发演示报告。

### 阶段 1: 品牌雷达最小可卖 MVP

目标：能卖给第一批客户。

- 品牌模板系统。
- 风险/机会评分。
- 日报和周报模板。
- 证据链展示。
- 飞书/企微/邮件稳定推送。
- 试点交付 SOP。

### 阶段 2: 付费试点与人工增强交付

目标：拿到付费和案例。

- 找 30 个目标客户。
- 为 10 个品牌生成免费样例报告。
- 约演示。
- 提供 999-1999 元/月的 30 天试点。
- 每天推送日报，每周人工校对周报。
- 用节省时间、发现风险、产出选题三个指标复盘。
- 转化到 4999 元/月增长雷达版。

### 阶段 3: 标准套餐与代理商版本

目标：提升客单价和复制效率。

- 多品牌项目管理。
- 白标报告。
- 批量日报/周报。
- 代理商折扣和分润机制。
- 客户外发版本。
- 人工校对工作台。

### 阶段 4: 私有部署与 500 万收入放大

目标：承接中大型客户和私有部署收入。

- 私有部署包。
- 权限、审计、备份。
- 数据源健康监控。
- 年度维护合同。
- 私有化交付清单。

## 11. 获客、留存与扩展

### 11.1 首批获客策略

首批不建议做大规模广告，而应做高密度试点：

1. 建立 30 个目标客户名单。
2. 手工为 10 个品牌生成免费样例报告。
3. 用样例报告约演示。
4. 提供低门槛 30 天试点。
5. 试点期间高频推送和人工校对。
6. 结束时用业务指标复盘价值。
7. 转化到标准套餐。

客户来源：

- 公关公司老板。
- 营销代理商负责人。
- 新消费品牌创始人。
- 本地生活品牌市场负责人。
- 内容代运营公司。
- 创始人 IP 服务商。

### 11.2 销售话术

不要说：

> 我们是 AI 舆情监控系统。

应该说：

> 我们每天帮品牌团队自动盯风险、看竞品、找热点选题，并把可执行建议推到你的飞书或企微。

更具体的话术：

```text
你们现在每天需要人工看微博、抖音、知乎、百度和行业媒体吗？
我们可以每天早上自动给你一份品牌舆情和营销机会日报：
有没有负面，竞品做了什么，今天有什么热点能借势，以及市场团队应该做什么动作。
先试 30 天，每天推送，周末给你一份人工校对周报。
```

### 11.3 留存和扩展

留存核心不是功能数量，而是客户是否每天打开报告并据此行动。

建议机制：

- 每周客户成功复盘。
- 报告采纳反馈。
- 选题被采用记录。
- 风险提前发现记录。
- 竞品动作月度总结。
- 模板持续优化。
- 从单品牌扩展到多品牌。

## 12. 关键指标

产品指标：

- Time to First Report: 从注册到第一份报告的时间。
- Report Open Rate: 报告打开率。
- Alert Precision: 提醒准确率。
- False Positive Rate: 误报率。
- Feedback Rate: 用户反馈率。
- Weekly Active Projects: 每周活跃项目数。
- Retention: 4 周留存。
- Source Health: 数据源健康率。
- AI Cost per Report: 单份报告 AI 成本。
- Action Rate: 客户根据报告采取动作的比例。
- Content Adoption Rate: 报告选题被采纳的比例。
- Risk Lead Time: 风险被系统提前发现的时间。

商业指标：

- 试用转付费率。
- 客单价。
- 月续费率。
- 私有部署成交周期。
- 品牌增长雷达版客户数。
- 代理商版客户数。
- 私有部署客户数。
- 每个品牌模板的转化率。
- 每个客户的配置交付时间。
- MRR。
- ARR。
- 500 万目标完成率。

## 13. 风险与应对

### 13.1 数据源风险

风险：第三方热榜、RSS 和平台接口不稳定。  
应对：数据源健康监控、多源冗余、失败降级、异常提醒、可替换数据源配置。

### 13.2 AI 成本风险

风险：AI 调用成本随报告量增长。  
应对：缓存、分层模型、小模型预筛、大模型精修、按套餐限制调用量。

### 13.3 报告质量风险

风险：报告泛泛而谈，不能形成业务动作。  
应对：行业模板、人工校对、反馈机制、证据链、动作清单和质量评分。

### 13.4 安全风险

风险：配置文件、API Key、客户数据泄露。  
应对：认证、权限、密钥加密、审计日志、备份、私有部署安全基线。

### 13.5 竞争风险

风险：大型舆情平台覆盖更广。  
应对：避开覆盖范围对比，强调中文热榜、低成本部署、AI 助理化、品牌模板和交付速度。

### 13.6 交付不可规模化风险

风险：人工增强过重，难以复制。  
应对：将人工校对沉淀为模板、规则、评分体系和 SOP，逐步产品化。

## 14. 优先级清单

### 14.1 紧急优先，1-2 周

- 登录认证和基础权限。
- API Key 与配置保护。
- 品牌样例报告。
- 品牌模板雏形。
- 试点客户名单。
- 演示环境和部署文档。

### 14.2 高优先，2-4 周

- 风险评分和机会评分。
- 日报/周报模板升级。
- 证据链展示。
- 数据源健康监控。
- 反馈系统。
- 试点交付 SOP。

### 14.3 中优先，4-8 周

- 项目化管理。
- 多品牌管理。
- 代理商白标报告。
- 人工校对工作台。
- 成本统计。
- 报告 PDF 导出。

### 14.4 低优先，8 周以上

- 重型多租户 SaaS。
- 支付系统。
- App 或小程序。
- 短视频总结。
- 成果分成系统。
- 大型企业复杂权限体系。

## 15. 关键讨论点

后续仍需持续讨论：

- 产品定位：是“舆情工具”还是“营销机会助理”？建议统一为后者。
- 定价策略：早期是否用低价试点换案例？建议可以，但必须限制周期。
- 客户获取：先直销还是先代理商？建议两条线并行，但代理商更适合快速复制。
- 团队资源：人工增强需要谁负责？建议先由创始团队直接交付，沉淀 SOP 后再外包或组建客户成功。
- 融资与商业模式：先证明 30-50 个高质量客户和代理商渠道，再考虑融资放大。

## 16. 竞品与差异化

主要竞品包括 Meltwater、Brandwatch、Mention、Talkwalker，以及国内舆情和营销监测系统。

TrendRadar / AiYX Data Radar 的差异化：

- 中文热榜和 RSS 整合更轻。
- 部署成本低，可私有化。
- 更适合中小品牌和代理商。
- 能用 AI 生成行动建议，而不是只给数据列表。
- 支持模板化交付。
- 可与飞书、企微、邮件、MCP 和未来 Agent 工作流连接。

核心竞争叙事：

> 大平台告诉你全网发生了什么；AiYX Data Radar 告诉你的品牌今天该做什么。

## 17. 最终建议

商业化路径应保持克制：

1. 不做泛化平台，先做品牌舆情与营销机会雷达。
2. 不靠低价订阅，先用高质量试点和人工增强验证价值。
3. 不先做复杂 SaaS，先做安全、模板、报告、交付和私有部署。
4. 不追求全网覆盖，强调中文热点、竞品动态、风险预警和行动建议。
5. 不把开放讨论中的所有方向同时推进，保留为 6-24 个月后的扩展选项。

近期最重要的动作是：完成可演示安全版本，准备 3-5 个品牌样例报告，约第一批代理商和消费品牌试点，用真实反馈打磨模板和价格。
